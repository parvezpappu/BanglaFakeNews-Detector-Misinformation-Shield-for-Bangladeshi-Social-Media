1. Project Goal

The goal of this project is to detect whether a Bangla news article is real or fake. The system accepts a Bangla headline and article content, predicts a label, shows confidence scores, checks trusted real-world sources, and stores prediction history.

2. Dataset Collection

Raw Bangla news datasets were stored in the local project under:

Dataset/
  Real/
  Fake/

The raw files were CSV files with inconsistent structures. Some files had headers, some used different column names, and labels were not always represented in the same way. To handle this, we created the dataset builder:

scripts/build_phase1_dataset.py

3. Dataset Cleaning

The dataset builder reads all CSV files from Dataset/Real and Dataset/Fake. It performs these cleaning steps:

  • Normalizes text by removing extra spaces and invisible characters.
  • Detects possible headline columns such as headline, title, and heading.
  • Detects possible content columns such as content, text, article, body, and news.
  • Detects possible label columns such as label, target, class_label, is_fake, and authenticity.
  • Skips rows with empty content.
  • Creates a stable duplicate key using normalized headline and content.
  • Removes duplicate news records.
  • Excludes conflicting duplicates where the same text appears with both real and fake labels.

4. Label Mapping

The system uses a binary label convention:

0 = fake
1 = real

For files in Dataset/Fake/, raw labels are interpreted as:

0 / fake / false -> fake
1 / real / true  -> real
empty label      -> fake

For files in Dataset/Real/, raw labels are interpreted as:

3       -> real
0, 1, 2 -> fake

5. Dataset Balancing and Splitting

After cleaning, the dataset is balanced so that fake and real classes have equal representation. Then the final dataset is split into:

80% training
10% validation
10% testing

BanglaBERT outputs probabilities for the two classes: fake and real.

8. Feature Engineering for LightGBM

The LightGBM branch uses a feature vector built from BanglaBERT outputs and handcrafted text features. The final feature count is 783.

  • 768 BanglaBERT CLS embedding features
  • 2 BanglaBERT probability features
  • 3 confidence features: max probability, probability margin, entropy
  • 10 basic text features: lengths, word counts, digit counts, punctuation counts, English character counts

The feature-building function is currently named build_xgboost_features(), but it is used by the LightGBM model in the deployed system.

10. Ensemble Model

The final prediction combines BanglaBERT probabilities and LightGBM probabilities. In the deployed backend, the configured ensemble weights are:

BanglaBERT weight = 0.50
LightGBM weight  = 0.50

The final class is the label with the highest combined probability.

14. Prediction Flow

  1. The user enters a Bangla headline and content in the frontend.
  2. The frontend sends the input to POST /predict.
  3. The backend loads EnsemblePredictor.
  4. BanglaBERT tokenizes and predicts fake/real probabilities.
  5. The BanglaBERT encoder extracts the CLS embedding.
  6. The backend builds the 783-feature vector for LightGBM.
  7. LightGBM predicts fake/real probabilities.
  8. The ensemble combines BanglaBERT and LightGBM probabilities.
  9. The backend returns label, confidence, final probabilities, and branch probabilities.
  10. The prediction is saved to MongoDB if history storage is configured.

15. Evidence Search

The system can check trusted Bangladeshi news sources. The deployed evidence endpoint uses:

app/backend/evidence_search.py

Search provider order:

  1. SerpAPI is used first if SERPAPI_API_KEY exists.
  2. Google Custom Search is used as fallback if SerpAPI fails and both GOOGLE_SEARCH_API_KEY and GOOGLE_SEARCH_CX exist.

Trusted domains include Prothom Alo, BDNews24, The Daily Star, Dhaka Tribune, Bangla Tribune, Jugantor, Kaler Kantho, Somoy News, BSS, Rumor Scanner, and BOOM Bangladesh.

16. History Storage

Prediction history is stored in MongoDB using:

app/backend/history_store.py

Each saved prediction includes:

  • headline
  • content
  • predicted label
  • confidence
  • final probabilities
  • BanglaBERT and LightGBM branch probabilities
  • created timestamp

19. Railway Runtime Model Loading

Large model files are not stored directly in the Railway build. Railway downloads them from Hugging Face at runtime using environment variables:

BANGLABERT_MODEL_NAME=ParvezPappu627/bangla-fake-news-banglabert
BANGLABERT_MODEL_SUBFOLDER=artifacts/banglabert_lightgbm_ensemble/banglabert_model
TOKENIZER_MODEL_NAME=ParvezPappu627/bangla-fake-news-banglabert
TOKENIZER_SUBFOLDER=artifacts/banglabert_lightgbm_ensemble/banglabert_model
LGBM_MODEL_REPO_ID=ParvezPappu627/bangla-fake-news-banglabert
LGBM_MODEL_SUBFOLDER=artifacts/banglabert_lightgbm_ensemble
LGBM_MODEL_FILENAME=lightgbm_model.joblib

We also added scikit-learn to backend requirements because the saved .joblib model needs it during loading.

20. Final System Output

For each submitted article, the system outputs:

  • Final label: real or fake
  • Final confidence
  • Fake probability
  • Real probability
  • BanglaBERT branch probability
  • LightGBM branch probability
  • Trusted-source evidence results
  • Prediction history entry

21. Short Summary

This project builds a complete Bangla fake news detection system. We clean and balance raw Bangla news datasets, fine-tune BanglaBERT, combine BanglaBERT embeddings and probabilities with LightGBM, deploy the model through FastAPI, serve the frontend through Netlify, load model artifacts from Hugging Face, verify news using SerpAPI or Google search, and store prediction history in MongoDB.
